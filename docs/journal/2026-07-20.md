## cross-lane drop reshuffle

Pilot friction from Michael: dragging a card from inbox to the bottom of horizon left the destination lane briefly mis-ordered at release, then everything animated into the intended positions — an awkward double movement. The mechanism: `handleDragOver` re-parents the card into the optimistic board at the index where it *crossed the lane border*, and same-lane hover after that is deliberately ignored — the moving gap under the pointer is pure dnd-kit transforms, with board state still holding the card at its entry slot. Release resets the transforms, so the entry-slot arrangement renders for a frame; the awaited `transition` response then repaints everything into place. The within-lane path never showed this because it repaints via `optimisticRanked` synchronously before awaiting — batched into the same render as the drag teardown, no intermediate frame.

The fix is the missing twin: `optimisticPlaced` (App.tsx, beside `optimisticRanked`) computes the drop from the pre-drag snapshot — card out of its origin lane, into the destination's ranked prefix at the anchored position — and `handleDragEnd`'s cross-lane branch now sets it synchronously before the await. The release frame shows the final arrangement; the server response matches it, so nothing moves twice. Unchanged by design: a drop into the unranked tail still serializes as end-of-ranked-list, so the card visibly snaps up past the unranked cards — but now once, at release, rather than as a post-response reshuffle.
